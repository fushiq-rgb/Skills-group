# 013-prd-agent：PRD需求工作流

## 核心设计理念

**门控机制**：关键节点强制用户确认，"继续"则下沉，"异议"则多轮沟通
**子模块独立**：各Agent可单独调用，无需跑全流程
**对话内确认**：所有征询通过Markdown清单在对话内完成，不跳转到其他界面

---

## 调用模式

### 模式一：全流程（复杂任务）

当用户说"开始PRD流程"、"启动PRD流水线"时，启用此模式：

```
用户输入原始素材
       ↓
  ┌─────────────┐
  │  Agent1     │  ← 执行
  │  会议转录整理 │
  └─────────────┘
         ↓
  ┌─── Gate1 ────┐          ← 对话内征询
  │ 需求完整性确认 │  用户确认"继续"或"有异议"
  └──────────────┘
         ↓
  ┌─────────────┐
  │  Agent2     │  ← 执行
  │  业务意图翻译 │
  └─────────────┘
         ↓
  ┌─── Gate2 ────┐          ← 对话内征询
  │ 业务目标确认  │  用户确认"继续"或"有异议"
  └──────────────┘
         ↓
  ┌─────────────┐
  │  Agent3     │  ← 执行
  │  PRD结构化写作 │
  └─────────────┘
         ↓
  ┌─── Gate3 ────┐          ← 对话内征询
  │ PRD方向确认   │  用户确认"继续"或"有异议"
  └──────────────┘
         ↓
  ┌─────────────┐
  │  Agent4     │  ← 执行
  │  评审挑战模拟 │
  └─────────────┘
         ↓
  ┌─── Gate4 ────┐          ← 对话内征询
  │ 评审结论确认  │  用户确认"继续"或"有异议"
  └──────────────┘
         ↓
    完整PRD文档交付
```

### 模式二：独立模块调用（轻量任务）

当用户单独调用某个Agent时，直接执行对应模块，无需走完全流程：

| 用户说法 | 调用模块 | 说明 |
|---------|---------|------|
| "整理需求"、"会议记录整理" | Agent1（Recorder）| 单独整理原始素材 |
| "翻译业务意图"、"拆解需求目标" | Agent2（Intent Translator）| 单独做意图翻译 |
| "写PRD"、"生成PRD" | Agent3（PRD Writer）| 单独写PRD初稿 |
| "评审PRD"、"过一下PRD" | Agent4（Reviewer）| 单独做评审挑战 |

## 门控指令设计规范

每次Gate输出结构化征询单，格式如下：

```
━━━━━━━━━━━━━━━━━━━━━━
【Gate N / 环节名称】

✅ 已完成：[产出摘要]
⚠️ 待确认：[具体问题]

请选择：
→ 回复「继续」：直接进入下一阶段
→ 回复「有异议」并描述：进入多轮沟通，我会逐一跟进
━━━━━━━━━━━━━━━━━━━━━━
```

**强制规则：**
- Gate输出前，上一Agent的所有输出项必须已写入飞书文档
- 不得用"自动继续"、"默认通过"等方式跳过Gate
- 多轮沟通可持续，直到用户明确说"继续"

---

## 飞书文档协作规则

**单文档分区域传递**，5个分区严格隔离：

| 分区 | 写入方 | 读取方 |
|------|--------|--------|
| 原始素材区 | 用户 | Agent1 |
| Agent1输出区 | Agent1 | Agent2、Gate1确认 |
| Agent2输出区 | Agent2 | Agent3、Gate2确认 |
| Agent3输出区 | Agent3 | Agent4、Gate3确认 |
| Agent4输出区 | Agent4 | 用户、Gate4确认 |

**禁止跨区读写**，违者杀无牢 💀

## 质量保障原则

- **零推测**：无来源依据的输出项不得写入文档
- **零遗漏**：强制输出项不得标注"待定"
- **零跨区**：严格遵守分区边界
- **Gate不跳过**：Gate是信任机制，不是流程装饰
- **完成确认规范化**：每Agent输出末尾含标准化完成声明
